# Internationalization

Every user-facing string in the package is a translation key under the
`console::` namespace, with English defaults in `resources/lang/en/`:

- `console.php` — Tools strings (status labels, interaction prompts, errors)
- `prompter.php` — Prompter exception/flow strings
- `validators.php` — validator messages

## Publishing and overriding

```bash
php artisan vendor:publish --tag=console-lang
```

This copies the language files into your application's `lang/` directory, where
you can edit them or add new locales (e.g. `lang/fr/console.php`).

## Adding a locale

Create the matching files under `lang/<locale>/` and set the locale:

```php
// config/console.php
'locale' => 'fr', // or null to follow the application locale
```

Strings are resolved with `__('console::validators.email')` and friends, with
`:placeholder` replacements. Exceptions resolve their messages the same way via
`ConsoleException::fromKey()`, falling back to a humanised key if a translation
is missing — so a missing key never produces an empty error message.

## Widget strings

User-facing widget text (the `Summary`, `Header`, `Menu`, `TaskProgress` and
`Callout` labels) lives under `console::console.widgets.*` and is resolved via
`Support\Lang`, which honours `config('console.locale')` **without** mutating the
host application's global locale. Each string keeps its English default as a
fallback, so widgets render correctly even outside a booted application. Override
them by publishing the lang files and editing `console.php`'s `widgets` array (or
adding a `lang/<locale>/console.php`).
